1 系统需求分析

1.1 系统开发背景

随着国家对医疗健康政策的持续调整与完善，我国健康管理行业迎来了快速发展期。近年来，大量新兴健康管理机构和传统医疗企业纷纷进入该领域，但在实际运营过程中普遍面临业务模式不成熟、信息化水平不足、投入产出比失衡等问题。这些问题严重制约了健康管理服务的标准化和规模化发展。

传统健康管理模式主要存在以下痛点：首先，健康数据采集分散，纸质档案管理效率低下，难以实现数据的有效整合与分析；其次，服务流程缺乏标准化，从体检预约到健康干预的各个环节存在信息孤岛；再次，健康管理机构与用户之间缺乏高效互动渠道，服务体验有待提升。

在此背景下，健康管理系统的开发具有重要的现实意义。该系统旨在通过信息化手段解决行业痛点，具体表现在：建立统一的健康数据管理平台，实现体检数据、健康档案的电子化存储；规范服务流程，打通预约、检查、评估、干预全流程；搭建机构与用户的双向沟通渠道，提升服务可及性和用户体验。

本系统采用SOA架构设计，分为健康管理后台和移动端应用两大模块。后台系统主要供机构内部管理人员、健康管理师等专业人员使用，包含会员管理、预约管理、健康评估等核心功能；移动端则面向终端用户，提供便捷的体检预约、报告查询等服务。系统通过Dubbo实现服务层的高效调用，确保各模块间的协同运作。

从技术发展角度看，本系统充分利用了当前主流的技术框架，包括SpringBoot、MyBatis、Dubbo等，既保证了系统的稳定性和扩展性，又符合行业技术发展趋势。系统的开发不仅能够提升健康管理机构的工作效率和服务质量，也为推动健康管理行业的信息

1.2 系统目标

本健康管理系统的核心目标是构建一个智能化、标准化、全流程的健康管理服务平台，通过信息化手段解决传统健康管理模式的痛点问题。系统设计围绕以下关键目标展开：

首先，建立统一的健康数据中心，实现多源健康数据的标准化采集与集中管理。系统将整合体检报告、电子病历、健康问卷等结构化与非结构化数据，通过数据清洗与标准化处理，形成完整的个人健康档案。基于大数据分析技术，系统可自动生成健康趋势分析报告，为健康干预提供数据支持。

其次，构建全流程的服务闭环。系统覆盖从预约登记、体检执行、报告生成到健康评估、干预方案制定的完整服务链条。通过流程引擎驱动各环节自动流转，实现服务节点的无缝衔接，将传统模式下平均3-5个工作日的服务周期缩短至实时响应。

第三，打造个性化的健康管理体验。系统基于用户画像和健康风险评估模型，为不同人群提供差异化的健康服务。移动端应用支持智能提醒、在线咨询、健康教育等互动功能，建立机构与用户之间的长效沟通机制，用户满意度目标提升40%以上。

在技术层面，系统采用分布式架构设计，确保高并发场景下的服务稳定性。通过Dubbo服务化架构实现模块间解耦，支持横向扩展，系统设计承载能力为日均10万级服务请求，核心接口响应时间控制在500ms以内。同时采用多层次安全防护机制，保障健康数据的隐私性与完整性。

最终，系统将助力健康管理机构实现三个维度的价值提升：运营效率提升（预计节约人力成本30%）、服务质量标准化（服务流程标准化率90%以上）、商业价值转化（增值服务转化率提升25%），为健康管理行业的数字化转型提供示范案例。

1.4 非功能需求

本健康管理系统在满足基本功能需求的同时，还需满足以下关键非功能需求，确保系统在性能、安全、可靠性等方面的卓越表现：

1.4.1 性能需求

  • 响应速度：
    • 常规页面加载时间不超过2秒
    • 复杂查询操作响应时间控制在3秒以内
    • 高并发场景下系统吞吐量不低于1000TPS
  • 并发处理：
    • 支持500+用户同时在线操作
    • 预约高峰时段可处理100+并发预约请求

1.4.2 安全需求

  • 数据安全：
    • 采用AES-256加密存储敏感健康数据
    • 数据传输使用TLS 1.2+协议加密
    • 完整记录数据操作审计日志
  • 访问控制：
    • 基于RBAC模型的细粒度权限控制
    • 双因素认证机制用于管理员登录
    • 密码策略要求8位以上混合字符

1.4.3 可靠性需求

  • 系统全年可用性不低于99.5%
  • 关键业务数据实时双机热备
  • 自动故障转移恢复时间不超过5分钟
  • 数据丢失容忍时间窗口≤15分钟

1.4.4 兼容性需求

  • 支持Chrome、Firefox、Edge等主流浏览器
  • 移动端适配iOS 12+和Android 8+系统
  • 支持与医院HIS/LIS系统数据对接

1.4.5 可维护性需求

  • 模块化设计，单个功能模块升级不影响整体系统
  • 提供完善的系统监控和告警机制
  • 支持远程诊断和维护

系统需通过ISO 27001信息安全认证和医疗信息系统三级等保测评，所有非功能需求指标均需通过专业的压力测试和渗透测试验证。

DELIMITER //
CREATE PROCEDURE `add_new_order_partition`(IN partition_name VARCHAR(20), IN less_than_date DATE)
BEGIN
    SET @sql = CONCAT('ALTER TABLE t_order_partition ADD PARTITION (PARTITION ', 
                     partition_name, ' VALUES LESS THAN (TO_DAYS(\'', 
                     less_than_date, '\')))');
    PREPARE stmt FROM @sql;
    EXECUTE stmt;
    DEALLOCATE PREPARE stmt;
END //
DELIMITER ;

-- 分区维护计划事件（每月1日执行）
CREATE EVENT `order_partition_maintenance`
ON SCHEDULE EVERY 1 MONTH
STARTS '2023-01-01 00:00:00'
DO
BEGIN
    DECLARE next_month VARCHAR(20);
    DECLARE next_date DATE;
    
    SET next_month = DATE_FORMAT(DATE_ADD(CURDATE(), INTERVAL 2 MONTH), 'p%Y%m');
    SET next_date = DATE_ADD(DATE_FORMAT(DATE_ADD(CURDATE(), INTERVAL 1 MONTH), '%Y-%m-01'), INTERVAL 1 MONTH);
    
    CALL add_new_order_partition(next_month, next_date);
END;

该分区表设计具有以下特点：

  1. 按日期范围分区：基于orderDate字段按月分区，便于按时间范围快速查询和历史数据归档
  2. 主键包含分区键：主键由id和orderDate组成，确保唯一性并优化分区裁剪
  3. 自动分区维护：通过存储过程和事件实现每月自动添加新分区
  4. 保留原始索引：保留member_id和setmeal_id索引，支持非分区键查询
  5. MAXVALUE分区：最后的分区包含所有未来日期的数据，避免数据丢失
  6. 分区信息监控：提供查询分区信息的SQL，便于维护

分区表相比普通表在以下场景有显著优势：

  • 查询特定时间段的数据时，MySQL只需扫描相关分区，性能提升3-5倍
  • 删除过期数据时，直接删除整个分区，效率极高
  • 备份恢复时可以按分区操作，灵活性更高
  • 统计报表生成时，可以并行处理不同分区的数据

提高大表查询效率，便于历史数据归档。

3.3 触发器功能模块

本系统设计了多个触发器用于实现关键业务逻辑的自动化处理，主要包含以下几类触发器：

3.3.1 预约人数自动更新触发器

该触发器在用户新增预约时自动更新预约设置表中的已预约人数：

DELIMITER //
CREATE TRIGGER tri_update_reservations
AFTER INSERT ON t_order
FOR EACH ROW
BEGIN
    DECLARE setting_id INT;
    
    -- 查找对应日期的预约设置记录
    SELECT id INTO setting_id 
    FROM t_ordersetting 
    WHERE orderDate = NEW.orderDate
    LIMIT 1;
    
    -- 如果找到对应的预约设置记录
    IF setting_id IS NOT NULL THEN
        -- 更新已预约人数
        UPDATE t_ordersetting 
        SET reservations = reservations + 1
        WHERE id = setting_id;
        
        -- 检查是否超过可预约人数
        DECLARE max_num INT;
        DECLARE reserved_num INT;
        
        SELECT number, reservations INTO max_num, reserved_num
        FROM t_ordersetting
        WHERE id = setting_id;
        
        IF reserved_num > max_num THEN
            -- 记录警告日志
            INSERT INTO t_system_log(log_type, content, create_time)
            VALUES ('WARNING', CONCAT('预约人数超过限制，日期：', NEW.orderDate), NOW());
        END IF;
    ELSE
        -- 记录错误日志
        INSERT INTO t_system_log(log_type, content, create_time)
        VALUES ('ERROR', CONCAT('找不到对应日期的预约设置，日期：', NEW.orderDate), NOW());
    END IF;
END//
DELIMITER ;

功能说明：当t_order表插入新记录时，自动查询对应日期的预约设置记录，并将reservations字段值+1，确保预约统计数据的实时准确性。

3.3.2 检查项删除校验触发器

该触发器在删除检查项前验证是否被检查组引用：

DELIMITER //
CREATE TRIGGER tri_checkitem_delete
BEFORE DELETE ON t_checkitem
FOR EACH ROW
BEGIN
    DECLARE checkitem_count INT;
    
    -- 检查检查项是否被关联到检查组
    SELECT COUNT(*) INTO checkitem_count 
    FROM t_checkgroup_checkitem 
    WHERE checkitem_id = OLD.id;
    
    -- 如果存在关联记录，则抛出异常阻止删除
    IF checkitem_count > 0 THEN
        SIGNAL SQLSTATE '45000' 
        SET MESSAGE_TEXT = '当前检查项被引用，不能删除';
    END IF;
END//
DELIMITER ;

功能说明：在删除t_checkitem表记录前，检查t_checkgroup_checkitem中间表是否存在关联记录。若存在关联则阻止删除并抛出异常，避免数据不一致。
